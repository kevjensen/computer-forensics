I put all the sha1 values and their respective file names into a file named sha1values.txt and then used the following command: uniq -D -w 30 sha1-sorted.txt

This printed all values with 40 characters the same next to each other to the command prompt. There I compared to see which of these were suspicious.

On the control the files for a majority of these were identical, with two exceptions:

/bin/bash and /bin/false

/usr/bin/unzip and /usr/bin/zipinfo

However, these files on the target *are* identical.

First, I studied what /bin/false does, and why it might be identical to /bin/bash and /bin/false "is just a binary that immediately exits, returning false, when it's called, so when someone who has false as shell logs in, they're immediately logged out when false exits" so when /bin/false was replaced with /bin/bash what might happen is that when soemoen has a false as shell logs, it actually does not do what /bin/false is supposed to do which is exit, and instead they can proceed to do things in the shell.

Second, /usr/bin/unzip and /usr/bin/zipinfo are identical. When interrogating the man page for zipinfo and unzip the definitions below are given.
